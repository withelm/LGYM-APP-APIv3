# LgymApi.Application.csproj

- Purpose: use-case and business orchestration layer.
- Contains: services, service interfaces, repository abstractions, application models, mapping core, notification abstractions, and module-owned app DI helpers.
- Rules: own business rules, authorization checks, transactions, and UoW commits here.
- Boundary: do not reference infrastructure implementations or either `LgymApi.BackgroundWorker*` project or namespace.
- Platform background-command ports: `IActionCommand` and `ICommandDispatcher` live under `Platform/Contracts/BackgroundCommands`; the 15 feature commands stay in their owning module `Contracts/BackgroundCommands` folders. Worker runtime types must not leak through these contracts.
- `ICommandOutboxWriter` is the stage-only background-command port for use cases that must commit a command envelope atomically with their own state; callers own the shared unit-of-work commit and do not schedule dispatch directly.
- Platform serialization contracts: the canonical persisted-payload `JsonSerializerOptions` and typed-ID converters live under `Platform/Contracts/Serialization`; API HTTP JSON reuses the typed-ID converter but keeps its stricter enum input policy.
- Durable command serialization writes the legacy `LgymApi.BackgroundWorker.Common.Commands.*` canonical ID and byte-compatible JSON. Application CLR full names are read aliases only; `CommandEnvelope.CommandTypeFullName` retains its legacy name while storing the canonical ID.
- `ICommandEnvelopeRepository.Detach` clears a failed pending insert before the dispatcher reloads the winning correlation-id row after a uniqueness conflict.
- DI ownership: feature helpers live here for business services only, and the platform/reference-data helper keeps shared app-level roots such as enum conversion and config services.
- Notifications: Application owns module-facing password scheduling, push delivery use cases, and provider-neutral push contracts. Worker supplies scheduling and job adapters, while Infrastructure supplies FCM delivery.
- Notifications push event, result, provider, retry-settings, and scheduling contracts are canonical under `Notifications/Contracts/Push`; Worker implements scheduling while Infrastructure owns the private FCM provider seam, including raw installation tokens.
- Notifications boundary: provider-neutral public contracts and event or intent models let source modules request or observe notification behavior without exposing entities, repositories, FCM, Hangfire, Worker, or Common types. Application has no Worker/Common dependency; see [`issue-381-notifications-boundary.md`](../docs/modular-monolith/issue-381-notifications-boundary.md).
- #381 documents ownership only: it does not change project references, endpoints, providers, schedulers, jobs, physical entity locations, the `AppDbContext`, or the migration stream.
- Training ELO scoring is selected by exercise profile in the training service; legacy exercise create/update methods always use the standard profile while privileged variants can opt into alternate formulas.
- Completed-training execution and history reads are owned by `WorkoutProgress/TrainingExecution`; `ITrainingService` remains the compatibility facade for existing Training API and coaching callers.
- `WorkoutProgress/ProgressData` owns immutable score-chart, measurement, main-record, and ELO read/write models behind `IWorkoutProgressReadWriteService`; legacy feature services remain compatibility facades and API mappings preserve legacy payloads.
- `WorkoutProgress/Dashboard` owns the public trainer-dashboard read facade. Coaching authorizes the trainer-trainee relationship before passing authorized IDs to it; its training details are explicit read models rather than Training, ExerciseScore, or Exercise entities.
- Workout & Progress cross-module access is limited to its `ProgressData`, dashboard, ranking, training execution/history, and `Contracts/ReportingIntegration` contracts/read models. The production `#386` integration exposes a Workout & Progress-owned idempotent measurement consumer; Reporting stages the separate Reporting-owned command, and legacy routes and payloads remain unchanged.
- Reporting submission acceptance derives measurement triples through a persistence-neutral Reporting factory and stages its accepted-progress command through `ICommandOutboxWriter` in the submission unit of work. The committed envelope is dispatched by the existing Worker ActionMessage flow; direct measurement persistence is outside the acceptance path.
- Accepted-progress operations are observable through event ID, report submission ID, correlation ID, causation ID, schema version, outcome, retry or dead-letter state, and counts. Raw answer JSON, photos, device tokens, and payload dumps are not logging fields.
- Training ELO formulas are implemented as one strategy class per `ExerciseEloFormula` under `Application/Common/Training/Elo` and resolved through DI instead of switch logic in `TrainingService`; the pull-up profile rewards lower weight with `PullupWeighted`.
- Enum lookups are the source of truth for front-end choice lists; `ExerciseEloFormula` is exposed through the enum lookup service and should not be duplicated as a hardcoded UI list.
- Enum lookup entries should carry a stable enum-string `id`; `name` and `displayName` should both be translated display text. Front-end choice lists should bind to the lookup `id`.
- Application input records for lookup-backed exercise forms should receive already-normalized enum values from API mapping profiles; do not parse lookup ids inside application services.
- Push installation lifecycle contracts and input models live in `Notifications`; their endpoint methods accept the authenticated `Id<User>` and current `UserSession`, commit after staging, and avoid leaking the Identity `User` entity. `IUserSessionTerminationService` consumes the explicit staging-only session-disassociation seam and retains its single commit without directly depending on the installation repository.
- Generic push enqueueing lives in `Notifications/PushNotificationService`; it creates installation-scoped durable push intents from privacy-safe payload contracts. The delivery-key unique index is the cross-instance idempotency authority; unique-conflict losers detach and reload the winner, while atomic durable scheduling reservations prevent duplicate scheduler enqueues.
- `Notifications/PushNotificationDeliveryService` owns durable push claim, delivery state transitions, retry scheduling decisions, and UoW commits. Post-claim exceptions and cancellation are persisted as bounded transient recovery states before propagation, so claimed messages are never stranded in `Sending`; scheduler failures leave the due delivery explicitly unscheduled for later enqueue recovery. Its provider port identifies installations by typed ID only, so Infrastructure resolves raw device tokens privately.
- `Notifications/InAppNotificationService` is the shared fanout point after a notification is durably saved: it publishes the existing SignalR in-app update and also enqueues a privacy-safe push event with the saved `InAppNotification` id plus `RedirectUrl` as the push deeplink.
- `Notifications/NotificationEventBridge` is the internal seam for backend event producers and the in-app fanout path; it forwards generic push events with IDs only and links the existing `InAppNotification` id for mobile lookup/routing.
- Notifications exposes six fact-only Coaching intents under `Notifications/Contracts/Events`; each selects exactly one eligible legacy channel, owns resource rendering, culture fallback, delivery metadata, in-app fanout, email identity checks, and provider-neutral email scheduling requests. `ICoachingEmailNotificationScheduler` carries those requests to the Worker adapter, while `ICoachingEmailNotificationFeature` exposes only the configured scheduling state. `ICoachingNotificationReadService` maps invitation persistence facts to the minimal Coaching-owned public fact for Worker adapters.
- `Notifications/StalePushInstallationCleanupService` tombstones inactive installations by `LastSeenAt` and preserves installation/message audit history instead of deleting old rows.
- Google account unlinking uses `IUserExternalLoginRepository` for active-row lookup plus soft-delete staging so the service layer can tombstone the current Google login without introducing provider-agnostic unlink behavior.
- Identity password recovery owns and consumes `IPasswordRecoveryEmailScheduler` and its seven-value typed request under `Features/PasswordReset/Contracts`; Worker/Common payloads remain outside that Application port, and the service supplies an explicit empty `ResetUrl`.
- Identity user workflows are split across `IUserRegistrationService`, `IUserCredentialLoginService`, `IUserSessionTerminationService`, `IUserProfileService`, `IUserRankingService`, `IUserAdminAccessService`, and `IUserRoleAdministrationService`. API callers use the focused login, session, profile, ranking, and admin contracts; `EloRegistryService` uses registration inside its existing outer transaction.
- Identity owns `IAccountReadService` for immutable active-account facts containing `Id<User> Id`, `Name`, `Email`, `Avatar`, `PreferredLanguage`, `PreferredTimeZone`, and `CreatedAt`, available by ID, normalized email, or ordered ID batch. Reads remain active-only; the ordered batch replays one `IUserRepository.GetByIdsAsync` read in caller order, retains duplicate input IDs, and omits missing or soft-deleted accounts (`null` for unavailable single reads). `AccountReadMappingProfile` owns every `User -> AccountReadModel` transformation. Coaching consumes the immutable `CreatedAt` fact only after identity enrichment to preserve trainer-dashboard `createdAt` sorting parity; this adds no API, route, or schema change. Notifications has no separate account-fact projection while its records remain ID/message based.
- The focused split preserves existing API route, HTTP verb, and payload contracts. `IUserRoleAdministrationService.UpdateUserRolesAsync` remains the focused role-administration use case, while the existing route-backed `IRoleService.UpdateUserRolesAsync` path remains unchanged.
- No monolithic `UserService`, `IUserService`, `IUserServiceDependencies`, or broad replacement dependency aggregate remains. Dependency aggregates, where needed, are owned by a single focused Identity service.
- Training Planning Plan operations are focused use cases under `TrainingPlanning/Plan/*`; controllers call their focused contracts, and Application inputs/results use typed IDs and read models. `IActivePlanPointerStore` remains the typed-ID seam for staging or reading the current user's active-plan pointer without accessing Identity entities or repositories.
- Training Planning owns seven immutable managed-plan contracts under `TrainingPlanning/Contracts/ManagedPlans`. The completed Coaching collaboration slices authorize through `ICoachingRelationshipAccessService` or active-link facts, then delegate to these owner contracts while preserving trainer-template creation, clone versus trainee activation, active-pointer, error, and transaction semantics. `ManagedPlanMappingProfile` is registered by the Application mapper scan and every `Plan` result projection uses injected `IMapper` with a fresh `MappingContext`.
- Coaching persistence is split into invitation, active-link, unpaged fact-reader, and trainee-note/history ports under `Coaching/Persistence`. Their models are immutable Coaching facts and typed identifiers only; the Application-scanned `CoachingPersistenceMappingProfile` owns all entity/fact and write-model/entity transformations. These focused ports are the sole Coaching persistence registrations; broad Coaching repository and service seams are retired.
- Coaching exposes `ICoachingRelationshipAccessService` under `Coaching/Contracts/Access` as the ID-only Reporting/Nutrition authorization surface. Its immutable decision distinguishes trainer status from active-link existence; the internal implementation composes Identity access reads with Coaching active-link persistence. Nutrition Diet Plans and Supplementation consume only this Coaching contract for relationship facts and retain their own established error adaptation, owner writes, commits, and Diet Plan notification timing. Reporting and Recurring Reporting consume the same published decision instead of `ITrainerRelationshipRepository`, map its facts to their existing Reporting errors, and retain photo self-access, report authorization, commits, notification timing, and accepted-progress outbox ownership.
- Training Planning owns and PlanDay consumes the ID-only `IPlanDayRelationshipAccessPort` under `TrainingPlanning/Contracts/PlanDay`; Coaching implements and registers its adapter, which projects `ICoachingRelationshipAccessService` to the active-relationship boolean without adding a Training Planning-to-Coaching dependency. PlanDay's owner short-circuit and owner-specific relationship authorization remain in the service, while its exercise/training repository dependencies remain unchanged legacy debt.
- Workout & Progress owns the ID-only `IMeasurementsRelationshipAccessPort` under `WorkoutProgress/Contracts/Measurements`; Coaching implements and registers its adapter, which projects `ICoachingRelationshipAccessService` to the active-relationship boolean without adding a Workout & Progress-to-Coaching dependency. `MeasurementsService` uses this port with `IUserAccessReadService` for its completed trainer authorization path; unrelated Workout & Progress persistence debt remains outside this access cutover.
- Coaching invitation creation, email creation, unexposed list, paginated list, and public-status operations are focused contracts under `Coaching/Invitations`. They use only typed IDs, Coaching facts, and Identity account facts; the paginated slice enriches unpaged invitation facts before filtering, sorting, counting, and paging. The anonymous public invitation API calls `IPublicInvitationStatusUseCase` directly, while the focused slice owns invitation lookup, bound-trainee detection, and Identity email-account resolution.
- Coaching invitation acceptance, rejection, and revocation are focused one-method contracts under `Coaching/Invitations`. Accept and reject validate a bound trainee ID or normalized email through Identity facts without mutation; an expired pending email invitation stages only `Expired` plus `RespondedAt`, stays unbound, creates no link, and queues no command. Valid pending email acceptance or rejection binds immediately with its staged response, while accepted replay, link-conflict mapping, revocation ownership, command timing, and UoW boundaries retain legacy behavior.
- Coaching relationship unlink, trainee detach, and current-trainer reads are focused one-method contracts under `Coaching/Relationships`. Their immutable inputs carry only typed user IDs; unlink authorizes through the Coaching relationship-access contract and sends no notification, detach commits the staged link removal before queuing the existing relationship-ended command, and current-trainer profile enrichment uses active Identity account facts plus the registered relationship mapping profile. Trainee relationship API actions call these focused contracts directly; the broad relationship facade and dependency bag are removed.
- Coaching trainer-dashboard reads combine complete unpaged Coaching facts with one ordered Identity account batch, capture one `now`, map the enriched logical rows through `TrainerDashboardMappingProfile`, and only then apply search, relationship/status filters, deterministic sorting, counting, and paging. Five focused `Coaching/Progress` contracts authorize through the Coaching relationship-access seam before one `IWorkoutProgressDashboardReadService` call and preserve its established read models and not-found translation.
- Seven focused `Coaching/ManagedPlans` collaboration contracts authorize trainer operations through `ICoachingRelationshipAccessService`; the trainee active-plan read verifies its Coaching active-link fact. Each slice maps its immutable typed-ID input through `ManagedPlanCollaborationMappingProfile`, delegates exactly once to the corresponding Training Planning managed-plan owner contract, and passes through the owner's result, error, clone, active-pointer, and transaction semantics.
- Coaching trainer-note list, create, update, delete, and history operations plus trainee-visible list and detail reads are focused one-method contracts under `Coaching/TraineeNotes`. Their immutable inputs carry trainer, trainee, and note typed IDs plus shared immutable upsert data; `TraineeNoteMappingProfile` owns fact, write, and read transformations. Trainee-visible reads use the existing visible, non-deleted, trainee-owned persistence semantics without relationship authorization or commits. Create, update, and soft delete stage the note and immutable history row before one owner `IUnitOfWork` commit, then create and update dispatch the existing notification command only after a successful commit under the legacy visibility-transition rules. Trainer and trainee note API actions call these focused contracts directly; the broad note facade is removed.
