# LgymApi.BackgroundWorker.csproj

- Purpose: Hangfire/background worker implementations for Application contracts and the bounded Common seam.
- Contains: persisted job implementations, the closed command runtime and handlers, scheduler adapters, and Worker DI.
- Rules: keep jobs idempotent where practical and register worker services in the worker module.
- Boundary: do not let the worker module become a second composition root for the API.
- Composition: worker startup uses application-only helpers where needed, especially the notifications seam, so it does not re-register module infrastructure.
- Durable commands: Worker owns the injected closed 15-command registry. It writes legacy Common command IDs as canonical IDs, accepts Application CLR full names only as read aliases, and does not scan loaded assemblies for command types.
- Runtime contracts: Worker owns the typed background-action resolver, which hides DI scopes behind exact handler-name lookup. Its DI helper registers one closed 15-row command registry, the 16 exact handlers, the Application dispatcher port, and validates handler cardinality after registration.
- `CommandOutboxWriter` implements Application's stage-only command-envelope port with the same registry, serialization, and deterministic-correlation rules as `CommandDispatcher`; it stages only and leaves commit and later committed-intent scheduling to the caller/runtime flow.
- Accepted report progress uses the production outbox path: Reporting stages `ReportSubmissionAcceptedProgressCommand` before its submission commit, then the existing committed-intent and `ActionMessage` infrastructure invokes the Workout & Progress consumer. The consumer owns measurement persistence and idempotent replay handling.
- Accepted-progress delivery treats `Applied` and `Duplicate` as success. Invalid, unsupported-schema, and poison outcomes become sanitized bounded failures for the existing retry/dead-letter path; unexpected consumer or persistence exceptions propagate for retry. Logs omit raw answer JSON, photos, device tokens, and payload dumps, while retaining event ID, report submission ID, correlation ID, causation ID, schema version, outcome, retry or dead-letter state, and counts.
- **Background Job Updates**: Added `[DisableConcurrentExecution(60)]` attribute to `EmailJob.ExecuteAsync` to serialize overlapping `EmailJob.ExecuteAsync` executions across worker instances. This reduces concurrent overlap, but exactly-once delivery still relies on the durable notification guard in `EmailJobHandlerService.ProcessAsync` (via `TryTransitionToSendingAsync`) because Hangfire locking is not keyed by `notificationId`.
- Push delivery uses `PushNotificationJob` plus the thin `PushNotificationJobHandlerService` adapter to invoke the Application-owned claim/send/state-transition/retry use case. `PushNotificationJob` retains `[AutomaticRetry(Attempts = 0)]` because Application persists and schedules deterministic recovery; invalid-token outcomes still tombstone the bound installation.
- Background-created in-app notifications flow through the Application-owned `NotificationEventBridge`, so they enqueue durable push intents after persistence in the API composition root. Testing retains that bridge while Worker selects no-op schedulers, preserving durable intent creation without Hangfire execution.
- Recurring job `push-stale-installation-cleanup` marks inactive installations with `DisabledReason=InactiveStale`; it is intended as rollout hygiene and should stay enabled even when outbound send delivery is feature-flagged off.
- `PasswordRecoveryEmailSchedulerAdapter` maps the Identity-owned seven-value request to the unchanged Common password-recovery payload and forwards cancellation to the generic email scheduler; the Worker DI helper owns both generic and Identity-port bindings.
- Push scheduling is selected once by the Worker DI helper: testing uses `NoOpPushBackgroundScheduler`, while non-testing uses `HangfirePushBackgroundScheduler`. Provider delivery remains Infrastructure-owned FCM.
- The same testing split selects no-op email and action schedulers; non-testing selects their Hangfire implementations. Common job interface targets, method signatures, and recurring IDs remain unchanged.
- Notifications boundary: Worker provides the compatibility adapters, job execution, and scheduler selection. It does not expose Hangfire or Common types through Application's provider-neutral notification contracts; see [`issue-381-notifications-boundary.md`](../docs/modular-monolith/issue-381-notifications-boundary.md).
- #381 preserves project references, provider behavior, endpoints, job and recurring identities, physical entity locations, persistence, and migration behavior.
