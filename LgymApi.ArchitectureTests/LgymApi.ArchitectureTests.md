# LgymApi.ArchitectureTests.csproj

- Purpose: Roslyn-based architecture guard tests.
- Contains: dependency direction checks, ID boundary checks, DI placement checks, feature layout checks, mapping checks, enum guards, UoW guards, persistence ownership guards, single-production-DbContext guards, and ownership documentation parsing guards.
- Rules: treat failures as architecture violations unless intentionally documented.
- Boundary: keep tests focused on structural rules, not application behavior.
- `PersistedEntityOwnershipCatalog.cs` is the executable source of truth for the 48 persisted entities and eight owner totals. `PersistedEntityOwnershipDocumentationTests` verifies the Markdown view has every catalog row exactly once and rejects missing, duplicate, and unknown-row fixtures.
- Architecture-debt allowlists are no-growth. Owner-only re-keying retains the same source and target identities; wildcard exemptions and re-keying to a new source or target are forbidden.
- Issue-380 guards enforce that Application references neither Worker project, that Common retains only its bounded job and email surface, and that Application contracts expose no Worker or Hangfire runtime types.
- Issue-381 guards parse the stable Notifications documentation rows and protect provider-neutral Application contracts, Notifications persisted ownership, Worker/Common runtime seams, provider/runtime leak rules, and compatibility invariants without requiring physical moves. The source is [`issue-381-notifications-boundary.md`](../docs/modular-monolith/issue-381-notifications-boundary.md).
- #381 guard coverage preserves project references, endpoints, job identities, providers, physical entity locations, the single `AppDbContext`, and the single migration stream.
- Composition guards verify the closed 15-command and 16-handler registry, Worker testing and non-testing scheduler selection, Infrastructure FCM ownership, canonical command-ID compatibility, typed persisted job arguments, and unchanged Hangfire job identities.
- Cancellation propagation guards allow only the exact repository reload and UoW saves inside `PushNotificationDeliveryService.RecoverFromPostClaimFailureAsync` to use `CancellationToken.None`; all other public async service persistence calls must propagate their caller token.
- #391 guards keep Workout & Progress repositories and EF configurations module-owned, require its explicit public read/write contract surface for foreign modules, verify contract DI registrations are unique and owner-scoped, and preserve no-tracking/projection shapes for ranking and progress reads. #386 guards reject Reporting direct measurement repository, entity, writer, and consumer dependencies while allowing only the Reporting-owned accepted-progress command staging path and the Workout & Progress consumer registration.
- Run the project suite with `dotnet test LgymApi.ArchitectureTests/LgymApi.ArchitectureTests.csproj --configuration Release --no-build`. The existing Todo 20 boundary matrix covered the Application Worker dependency and Common surface guards.
- Issue-389 PlanDay authorization guards verify the implemented consumer-owned Training Planning port uses typed user IDs and its Coaching adapter remains the sole relationship bridge. Target Roslyn fixtures reject reintroduced direct Coaching relationship-service or repository dependencies; the unrelated PlanDay-to-Workout repository debt remains separately tracked.
- Issue-389 Measurements authorization guards verify the implemented consumer-owned Workout & Progress port uses typed user IDs and its Coaching adapter remains the sole relationship bridge. Target Roslyn fixtures reject reintroduced direct Coaching relationship-service or repository dependencies.
- Issue-389 Nutrition authorization guards require Diet Plans and Supplementation to consume only `ICoachingRelationshipAccessService`; their migrated services may not regain direct relationship or role repositories, Coaching persistence, or private Coaching implementations.
- Issue-389 Reporting authorization guards require Reporting and Recurring Reporting dependency bags, production services, and focused test factories to consume only `ICoachingRelationshipAccessService`; direct relationship repositories and other Coaching internals are rejected.
- Issue-389 Coaching guards require focused public contracts, owner-scoped registrations, consumer-owned relationship-access ports, and isolated Coaching persistence adapters. They reject restored broad Coaching repository or service seams and forbidden foreign persistence dependencies. The enforced boundary-debt baseline is 157; run the complete suite with `dotnet test LgymApi.ArchitectureTests/LgymApi.ArchitectureTests.csproj --configuration Release --no-build`.
